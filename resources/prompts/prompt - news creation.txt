# Context
- **Domain / topic:** Generating concise, date-based “News & Updates” entries from unified diffs of documentation/static site content.
- **Audience & use case:**
  - Primary audience: end-users and stakeholders reading a public “News & Updates” or “Changelog” section.
  - Secondary audience: documentation maintainers who need a clear, page-first summary of changes.
- **Available inputs / sources:**
  - A unified diff (git-style) comparing old and new site content (typically Markdown files).
  - Optionally, a **DATE override** in `YYYY-MM-DD` format.
  - Optionally, mappings from file paths to page URLs and/or commit/PR URLs and link texts, included alongside or within the diff.
  - Optionally, the current news page as a style reference, available at https://raw.githubusercontent.com/Health-RI/semantic-interoperability/refs/heads/main/docs/news.md.
  - At runtime, the caller will append an input block of the form:
    - `INPUT`
    - `DIFF:`
    - `<PASTE THE UNIFIED DIFF HERE>`
- **Key definitions or scope notes:**
  - A **page** is a content file (e.g., `docs/sssom-mapping-set.md`).
  - The “News & Updates” entry is **not** meant to expose a technical diff to users; it must instead highlight only important, user-relevant modifications or additions to the website. Minor or low-impact changes must **not** generate news items unless explicitly marked as newsworthy in the input (e.g., via a specific note or flag).
  - **Classification of pages:**
    - **Created**: page exists only on the new side (added file or near-total addition).
    - **Removed**: page exists only on the old side (deleted file or near-total deletion).
    - **Changed/Modified/Fixed**: page exists on both sides with meaningful content changes.
  - **Page Title derivation:**
    - Prefer the Markdown H1 `# ...` from the **new** file.
    - If missing, use the filename stem (strip directories and extension, replace dashes/underscores with spaces, capitalize words; e.g., `docs/sssom-mapping-set.md` → `SSSOM Mapping Set`).
    - For **Removed** pages (no new file), use the old file’s H1 if available, otherwise the old filename stem using the same rules.
  - **Meaningful content change:** changes to actual wording, structure, or content that are relevant for users (e.g., new sections, updated guidance, reworked examples). Ignore purely formatting-only edits (whitespace, line-wrapping, trivial Markdown syntax changes) and other small technical or cosmetic changes unless they are the **only** change for that page **and** have been explicitly requested to appear as news.
  - All references to “today” mean **today’s date in UTC**.

# Goal
- **Primary outcome:**

- **Success criteria:**

  - Group all bullets for the diff under this single date heading (one date section per run).
  - Include **one bullet per page** that has a Created/Removed/Changed status and a **user-relevant** change, with no duplicate bullets for the same page.
  - Correctly classify pages as:
    - **Created** when added or almost entirely new.
    - **Removed** when deleted or almost entirely removed.
    - **Changed/Modified/Fixed** when content is updated but page exists on both sides.
  - Use an initial **Verb** (no brackets) from: `Created`, `Changed`, `Modified`, `Fixed`, `Added`, `Removed`, `Renamed`, chosen to best describe the change.
  - Derive **Page Titles** according to the rules in Context; do not invent titles unrelated to the file or headings.
  - Summarize the **meaningful content change** for each page in **one short sentence**, ignoring formatting-only or minor internal edits unless they are the only change and explicitly requested as news.
  - Format each bullet on a **single line** as:
    `- **Page Title** — Verb: one-sentence description. ([link_text](link_url))`

    - If no URLs are provided, omit all links and commit/PR mentions.
  - Never fabricate links, pages, filenames, or commit/PR references not supported by the diff or explicit metadata.
  - Produce exactly **one** Markdown fenced code block, with no text before or after it.
- **Deliverables / format:**
  - Deliverable: **one Markdown fenced code block** (e.g., \```markdown … \```), containing:
    - A single date heading (`## YYYY-MM-DD`).
    - One bullet per relevant page, formatted exactly as specified above.

  - Internally ensure that all Success Criteria are satisfied; the output itself must be only the required code block.
